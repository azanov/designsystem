How to write localization-friendly strings

In order for localization to be carried out efficiently, developers must design UI so that localization is straightforward and localized content does not break the look and feel of the program.

  • Unicode! Your displayable text. Your file formats. Your database. Your everything. If it moves, it is Unicode. If it doesn't, it is still Unicode.
  • Stick to the industry standards. Don’t create a special file format.
  • DON’T re-use resource IDs. Delete the old ID and create a new one.
  • Do not hardcode anything related to text or locale related info.
  • Leave enough space (minimum of 30 percent additional room) for text expansion in any control or group of controls.
  • Text should dynamically wrap if it is going to be truncated.
  • Avoid UI Controls within sentences.
  • Avoid Hidden or Overlapped UI Controls.
  • Avoid placing button text into a string variable.
  • Beware of Gender-Specific Roles and Ethnic Stereotypes in Other Cultures.
  • Avoid Religious References, Political Symbols and Text In Graphics.
  • Do not be vague what you are referring to in the labels.

Many localizers are not programmers and might delete important details, such as closing quotation marks or semicolons, or they might translate programming keywords as well as strings. Then developers or localization engineers must take time to clean up these files so that they will compile correctly. Localizers use Translation Memory software to reuse already translated items from the previous project. Translation Memory software only supports industry standard file formats. In addition, hard-coded strings are difficult to find because they do not show up until the software is localized.
