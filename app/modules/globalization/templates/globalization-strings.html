How to write localization-friendly strings

String separation

DO NOT mix translatable strings and non-translatable strings in the same file.

  • If you include a translatable string in the resource file to be translated, it will be translated. Non-translatable strings must be kept in a separate file and not sent for localization. Example: SELECT is used in a SQL statement and should not be translated, so it should be kept in a separate non-translatable resource file.
  • If a non-translatable term appears in a translatable string, please include the term in a glossary and mark it as “Non translatable”.
    Example: Enterprise Manager is a product name and should not be translated. It should be listed in a glossary as “non-translatable term”. However, Enterprise Manager will show up as part of the translatable string.

Use of Upper Case

DON’T use all upper cases if words need to be translated

  • DATE FORMAT will not be translated unless it’s a heading
  • Date Format will be translated
  • DateFormat will not translated: never use CamelCase for terms that need to be translated. They will be left untranslated by default by any translator

Nouns vs. Verbs

Be clear whether you mean Noun or Verb. Some strings can be ambiguous. For example, View File can be interpreted as Noun or Verb. If you mean this to be a verb, please say “View the file” instead. Also, it is a good idea to leave a comment if possible.

Concatenated Words

DON’T concatenate words if they need to be translated. Concatenated words are always considered as non-translatable by default by translators.

  • DateFormat will not be translated.
  • Date_Format will not be translated.
  • Date.Format will not be translated.
  • Date Format will be translated.

Concatenated Sentences

Many translators have spent sleepless nights trying to decipher what a creative developer is attempting to convey to the user. There are at least as many different ways to create strings of text as there are developers.

Strings must be easy for localizers to comprehend. Requirements include:

  • No string concatenation at runtime.
  • When variables are necessary, use unique names.
  • Do not compound several variables.
  • Keep sentences in a single string.

Microsoft has a detailed discussion of the above requirements.

Example:

char szString[] = "I saw a "; char szFinalString[cbMaxSz]= szString + szObject + "?";

The translator sees in the resource file is "I saw a " and has no idea what follows “a”. In some languages nouns have gender: for instance, in German nouns can be masculine, feminine, or neuter, which has an influence on the sentence’s syntax and grammar. Which means the translator must make a choice, which will not fit all possible cases.

Sorting Order

Each language has its own sort order and the order is often very different from English. Your code must support all required sorting orders and behave appropriately when strings are compared.

Line/Word Breaking

Often sentences will have to wrap in mid-word if they are too long to fit in the provided space. In English and other Latin-based languages it is easy to find natural break-points between syllables using some basic parsing techniques. For non-Latin languages the natural breaks are not so easy to determine. Plus some languages use a character other than the standard period ‘.’ for an end-of-sentence designator (ex. Japanese – ‘。’). Some languages do not even have spaces between words. This makes parsing for natural line-breaks even trickier.

The code must be aware of line and word breaking rules for all required languages. It cannot conveniently base line–breaking and word–wrapping algorithms on a space character or on standard hyphenation rules. When multilingual text is to be parsed or displayed, special attention should be paid. Latin script follows some straightforward rules such as breaking a line at a space, tab, or hyphen. Word breaking is based on grammatical analysis and on word matching in dictionaries during text processing at run time.

Capitalization, Uppercasing and Lowercasing

Some languages do not have a one–to–one mapping between their uppercase and lowercase characters. Most non–Latin scripts do not even use the concept of lowercase and uppercase, as in the case of Chinese, Japanese, and Korean; Arabic, Farsi, Hebrew and Thai. The program must consider handling of linguistic nuances of capitalization, uppercasing and lowercasing. Characters should be handled correctly regardless of keystrokes associated with them.

Also, there are a number of uppercase letters with accents (É, Ý, Ñ, Ð, Ø, Å, …). You must ensure your product supports correct display.
