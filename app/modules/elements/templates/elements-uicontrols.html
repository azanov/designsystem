Input Fields

The form field is our most often used input, and thus has a range of states and uses that inform our users. It is our rule that the content within a field is the primary design focus, while the label is secondary.

Basic Form Fields

We have three sizes of most inputs: default, small, and large. The default size should be used for fields in all instances except:

  • If you have a large, crowded form (20+ fields) consider using the smaller size inputs
  • If you have an extremely light form (<10 fields) consider use large fields (Example: sign in screens)
  • Note that the field itself and its contents are smaller, but the label remains 14px always

Form fields have four primary states, each defined by its form and type coloring:

  • Default
  • Active
  • Disabled
  • Error

Some fields require some explanation to users so they understand what they’re being asked for. In these cases you can include a short line of instructional text below the field.

Should that not be sufficient, you can provide a small popover next to the label that provides additional context for users.

A password strength meter tied to the input field is specifically on registration pages to help users choose a secure, system-OK combination.

Writing Form Field Labels

When writing your labels and in-field prompt text, it’s important to stay consistent across your entire product. Here are some basic guidelines for how to write these critical pieces of user interface copy.

Style

  • Use short, active words and phrases.
  • Structure labels to avoid ambiguity in language.
  • To avoid confusing users, use consistent terminology for a given concept (e.g., “Input Recipient’s Name” vs. “Input Addressee’s Name” vs. “Input Contact’s Name”).

Capitalization

  • Form field labels use sentence capitalization, except when they include words or phrases that are acronyms or proper nouns (e.g., “RMA number,” “Cost Account,” “ZIP Code”).

Punctuation

  • Form field labels don’t conclude with punctuation, such as periods, colons.
  • Don’t phrase your form field labels as questions.

States

Dropdown menus have four primary states, each defined by its form and type coloring:
  • Default
  • Focused
  • Active
  • Disabled
Which states a user sees depends on whether or not the menu is enabled or disabled.

Defaults

When populating menus, we have certain rules:

  • Never present the user with a blank menu.
  • In most cases, “Select” should be your first menu item, to help prompt the user to make a selection.

The exception is when you are confident that a user will want to select a particular menu item at least 80% of the time. In that case, code your menu to preselect that item.

Sizes

We have three sizes of dropdown menus: standard, large and small. Use the standard size for all dropdowns except when:

  • You have a very crowded form (20+ dropdowns), in which case consider using small dropdown menus.
  • You have only a few dropdowns (<10), in which case consider using large dropdown menus.

Note that even when using large or small dropdown menus the label text remains the standard size.

Writing Checkbox Labels

When writing your labels, be descriptive enough so users understand what checking the box will do, but try to avoid very long labels. One way to do this is by introducing the checkboxes with a descriptive sentence or two that lends context to the checkboxes’ purpose.

Here are some basic guidelines for how to write these critical pieces of user interface copy.

Style
  • Use short, active words and phrases.
  • Structure labels to avoid ambiguity in language.
  • To avoid confusing users, use consistent terminology.
  • Use the same style and structure for all your labels. If one label is written as a complete sentence then write all of your labels as complete sentences rather than mixing and matching sentences with fragments or phrases.
Capitalization
  • Checkbox labels use sentence capitalization, except when they include words or phrases that are acronyms or proper nouns (e.g., “RMA number,” “Cost Account,” “ZIP Code”).
Punctuation
  • Checkbox labels don’t conclude with punctuation, such as periods or colons.
  • Don’t phrase your checkbox labels as questions.

Radio buttons

We have two sizes of radio buttons: standard and enhanced. In most cases, you should use the enhanced size. If your product has a lot of radio buttons, however, you should consider using the standard size to save space.

Note that if you just need to give the user a means to turn on or off a function, a toggle control is a better choice.
